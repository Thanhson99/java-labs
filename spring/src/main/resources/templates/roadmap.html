Phase 1: Java Core

You should finish this phase before trying to reason about Spring service flow.

Java variables, parameters, and methods

Goal: read method signatures without hesitation.

Read: LearningApp, cheat sheet sections for variables and methods.

Done when: you can explain parameter vs argument and write a method from scratch.

Control flow and loops

Goal: understand how Java branches and repeats work.

Read: control flow examples and loop-related tests in basic/.

Done when: you can replace one loop style with another correctly.

Classes, records, encapsulation, and OOP

Goal: know when to use class, record, enum, and interface.

Read: BankAccount, Student, DTO records in Spring.

Done when: you can design a small domain model without mixing every responsibility together.

Collections, streams, and analytics logic

Goal: transform in-memory data safely and clearly.

Read: analytics helpers and tests using collections and streams.

Done when: you can filter, map, sort, and group without copying examples blindly.

JDBC and database access

Goal: understand what Spring is abstracting away from plain SQL access.

Read: JDBC examples in basic/, then compare to JdbcTemplate usage in Spring.

Done when: you can explain connection, SQL execution, row mapping, and error handling.

Unit testing in Java

Goal: use tests as the fastest reading tool in the repo.

Read: JUnit tests in basic/src/test/java.

Done when: you can change code intentionally and predict which test should fail.

Phase 2: Spring Foundations

Here you move from Java syntax to a running backend application model.

Spring MVC controllers and templates

Goal: follow request flow from browser to controller to response.

Read: WebController, RegistrationController, public templates.

Done when: you can add one small public page or endpoint yourself.

JWT auth, refresh tokens, and sessions

Goal: understand why access token and refresh token are different.

Read: auth controller, token service, refresh token store, security config.

Done when: you can explain login, refresh, logout, logout-all, and reuse detection in order.

Transactions, rollback, and Flyway

Goal: see how one service method spans repository, audit, analytics, and rollback.

Read: transaction tests and Flyway migration files.

Done when: you can explain what gets rolled back and why.

Spring Boot testing with MockMvc

Goal: verify HTTP behavior without manually clicking everything in a browser.

Read: controller tests in the Spring module.

Done when: you can add a new route and test it first or immediately after.

Phase 3: Data And Integration

This phase is where the app starts to look like a real backend system.

Outbox pattern and durable event publishing

Goal: understand why DB write and event publish should not be coupled naively.

Read: RegistrationService, OutboxEventStore, OutboxDispatcher.

Done when: you can explain pending, published, retry, and dead-letter states.

Kafka and RabbitMQ producers and consumers

Goal: compare two transports without confusing business logic with transport glue.

Read: publishers, listeners, topology config, and notification service consumers.

Done when: you can say what is shared and what is transport-specific.

Microservice boundaries and notification service

Goal: think in boundaries, not only classes.

Read: main Spring app and extracted notification-service side by side.

Done when: you can describe why the service was extracted and what contract it consumes.

Observability and admin operations

Goal: read the system like an operator, not just a coder.

Read: actuator config, metrics class, system overview controller, admin page.

Done when: you can use runtime pages to explain what the app is doing now.

PostgreSQL integration with Testcontainers

Goal: trust the system against a real database, not only H2.

Read: PostgresContainerRegistrationTest.

Done when: you can explain why the project keeps both lightweight and realistic tests.

Phase 4: What To Build Next

These are not yet complete learning stages. They are your next serious upgrades.

Redis cache and distributed rate limit

Goal: move shared state out of one app instance.

Practice: add Redis-backed rate limiting and cache for repeated lookups.

Done when: multiple app instances could share the same limit state safely.

Persistent user accounts and role model

Goal: replace demo users with real database-backed accounts.

Practice: user table, password hashing, role table, registration flow.

Done when: auth no longer depends on in-memory demo credentials.

Exercise mode and guided tasks

Goal: turn the repo into practice, not just reading.

Practice: small tasks, expected outputs, and matching tests.

Done when: you can use the repo as a self-paced Java/Spring lab.

Study Rules

  1. If a topic is new, read the cheat sheet first.
  2. Then read the matching Java code and tests.
  3. Then change one behavior and rerun the test suite.
  4. Only then mark the topic complete.

Definition Of Done

  1. You can explain the topic without reading the code.
  2. You can point to the exact file that demonstrates it.
  3. You can change one behavior and predict the test impact.
  4. You can say where this concept appears in real backend work.
